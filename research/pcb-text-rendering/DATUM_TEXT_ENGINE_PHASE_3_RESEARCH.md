## Executive Summary

- **Verdict on the user's working hypothesis: confirmed in full.** The
  current pipeline lowers a `BoardText` to thousands of anonymous
  `BoardGraphicPrimitive { primitive_kind: "polygon" }` fragments at
  the protocol boundary (`board_text_graphics` in
  `crates/gui-protocol/src/lib.rs:3786`). Each fragment is a 4-vertex
  scanline trapezoid emitted by `resolve_contour_set_non_zero_scanline`
  (`crates/engine/src/text/outline.rs:309`). The renderer rebuilds
  quads from these via `push_board_graphic_primitive_world`
  (`crates/gui-render/src/lib.rs:4168`), which is the polygon-confetti
  path. Text identity is destroyed at the protocol boundary; the
  renderer cannot reconstruct it. The hypothesised correct architecture
  (semantic → layout → grouped TextGeometry → mesh/batch → renderer)
  is the right shape.
- **Recommended GPU representation: glyph-mesh cache with per-text
  instanced batching (CPU-tessellated triangles, cached by
  `(font_id, glyph_id, tolerance_class)`), with a screen-space glyph
  atlas as a deferred Phase 3.2 acceleration for high-density zoom-out
  views.** Atlas-only is wrong for a CAD tool that must zoom from
  full-board to glyph-level detail with deterministic goldens; pure
  vector-on-GPU (Vello) is too immature and too compute-heavy for the
  current wgpu render path; pure SDF artifacts on thin PCB silkscreen
  strokes. The mesh-cache + instancing model is what `lyon`-based
  systems converge on and it preserves the same world-space coordinate
  determinism that the routing kernel and DRC already assume.
- **Recommended new protocol type: `BoardTextGeometry` per text
  object** — Option C (hybrid) from Area 5: one protocol message per
  text UUID, payload is a glyph-instance batch (cached glyph IDs +
  per-glyph world transforms) plus the semantic `BoardTextPrimitive`
  for hit testing and selection. Identity is preserved at the protocol
  level; batching happens inside the renderer per layer/draw-pass.
  Option A (single per-text payload, no batching) is too coarse for
  the renderer; Option B (per-layer batch, identity in metadata) is
  fragile because metadata is easy to drop on the renderer floor.
- **Where tessellation lives: engine, with a content-addressed glyph
  mesh cache.** The engine is already the source of deterministic
  geometry for export (Gerber, ODB++ later). Putting tessellation in
  the engine makes one source of truth for renderer + export,
  preserves the existing determinism contract, and lets the cache key
  `(font_id, glyph_id, tolerance_class)` amortize across every text
  using the same glyph. The protocol adapter and renderer become thin
  consumers of pre-tessellated meshes — they no longer recompute.
- **Biggest implementation risk: `lyon` tessellator non-determinism
  across host architectures.** Phase 2 named float drift in Bézier
  flattening as the top determinism risk. Phase 3 inherits that risk
  and amplifies it: the tessellator's triangle output (vertex order,
  midpoint subdivision) is a second floating-point stage downstream of
  flattening. Mitigation: pin x86_64-linux + Mesa as the golden
  authority (already established by visual-regression research),
  cache by `(font_id, glyph_id, tolerance_class)` rather than
  `(font_id, glyph_id, exact_tolerance_nm)` to avoid one-pixel cache
  thrash, and keep the integer-arithmetic stroke (Newstroke) path
  intact as the canonical M7 baseline while the outline-mesh path
  ships behind a flag for the first slice.
- **Polygon-confetti reduction expected from Phase 3.0: roughly
  100×–300× fewer renderer draw items.** The repro fixture today
  produces 347,154 quads / 2,082,924 vertices and takes 1,698 ms to
  build. Phase 3.0 (mesh cache + per-text batching, no atlas) targets
  ~1,000–3,500 batched draws per board (one per text object plus
  per-layer batching), with retained-scene construction in the
  ~50–150 ms band on the same fixture. Cache-hit rate on the second
  build of the same scene should approach 100% on glyph meshes (only
  per-text transform matrices change).
- **Recommended Phase 3.0 first slice (concrete, ~1 working week of
  engineering):** introduce `eda_engine::text::TextGeometryBatch`
  (per-text-object grouped output: text_uuid, layer, world transform,
  ordered list of `(cached_glyph_handle, glyph_origin_world)`), plus
  an engine-owned `GlyphMeshCache` keyed by
  `(font_id, glyph_id, tolerance_class)`. Add a sibling protocol type
  `BoardTextGeometry` carrying the batch. Replace
  `board_text_graphics` (currently emitting `BoardGraphicPrimitive`
  fragments) with a `board_text_geometry` adapter that emits one
  `BoardTextGeometry` per text. Add a renderer consumer that builds
  one quad batch per layer using the cached glyph meshes. Keep the old
  path under a feature flag so the visual goldens migrate one fixture
  at a time, starting with `text-density-repro`. Do **not** ship glyph
  atlas in Phase 3.0; that is Phase 3.2.
- **Coexistence with current `render_cache` work**: Phase 3 is
  orthogonal. M7-IMP-014's correctness goal is "imported and native
  text land on the same Datum-owned generator regardless of KiCad
  cache presence." Phase 3 changes the *output representation* of
  that generator (grouped `TextGeometry` instead of per-fragment
  polygons). The two streams of work compose naturally: M7-IMP-014
  finishes the input-side normalization; Phase 3 fixes the output-side
  representation; both deliver into the same `BoardTextGeometry`
  protocol type.

## The Problem (verbatim user statement)

> Datum's current outline text path is producing visually correct
> text, but the internal representation is structurally wrong for
> interactive CAD performance. The `text-density-repro` fixture proves
> the issue: retained-scene construction takes 1,698 ms / 347,154
> quads / 2,082,924 vertices, because text geometry is being emitted
> as hundreds of thousands of tiny `BoardGraphicPrimitive` polygon
> fragments. This is "polygon confetti." It is not an acceptable
> long-term representation for a serious EDA text system.

## The Working Hypothesis (verbatim user statement)

> The likely correct architecture is:
> ```
> BoardText semantic object
>         ↓
> Engine text layout / font shaping
>         ↓
> Grouped TextGeometry object per text UUID
>         ↓
> Tessellated TextMesh or contour batch
>         ↓
> GUI retained scene batch
>         ↓
> Renderer draws one/few ranges per text object
> ```
> The current architecture effectively does this instead:
> ```
> BoardText semantic object
>         ↓
> Engine outline generation
>         ↓
> hundreds/thousands of anonymous board graphic polygons
>         ↓
> generic board graphics renderer
>         ↓
> massive retained quad buffer
> ```
> The second path is the problem. Research should validate or correct
> the working hypothesis and return an implementation-ready
> recommendation.

## Non-Negotiable Constraints (verbatim, treated as architectural floor)

1. Do NOT propose falling back to stroke fonts for performance.
2. Do NOT propose degrading visual fidelity to solve performance.
3. Do NOT propose treating every text fragment as a generic
   `BoardGraphicPrimitive`.
4. Do NOT make renderer behavior depend on KiCad `render_cache`.
5. Do NOT split imported text and native text into divergent
   rendering paths.
6. Do NOT move the burden onto the GUI renderer to reconstruct text
   semantics from anonymous polygons.
7. Text object identity MUST survive through the pipeline.
8. Hit testing MUST remain object-level, not fragment-level.
9. Datum owns the text engine.

These nine items are floor, not subject to evaluation. Every
recommendation in this report is filtered through them. A compliance
map appears near the end.

## Current Implementation: Where Confetti Is Born

To make the recommendations grounded against actual current state, the
key code paths today (read-only Phase 3 recon — no edits made):

| Stage | File:line | What happens |
|-------|-----------|--------------|
| Semantic | `crates/engine/src/board/board_info.rs` (`BoardText`) | UUID-bearing semantic object: text, position, rotation, layer, height_nm, render_intent, family, style, alignment, mirror, keep_upright, line_spacing_ratio_ppm, italic, bold, style_class. This layer is correct. |
| Engine layout | `crates/engine/src/text/layout.rs:layout_text_geometry` | Routes through `default_backend_for_attributes` → either `newstroke_backend` (stroke) or `vendored_outline_backend` (outline). Returns `Vec<TextGeometryPrimitive>`. |
| Engine outline | `crates/engine/src/text/outline.rs:layout_text_outline` | Calls `flatten_glyph_from_face` per glyph (Bézier → polyline via recursive midpoint subdivision, tolerance from `outline_flatten_tolerance_nm`), then `flattened_contours_to_contour_set` → `resolve_contour_set_non_zero_scanline`. |
| **Confetti generator** | `crates/engine/src/text/outline.rs:309` `resolve_contour_set_non_zero_scanline` | Walks every distinct y-value in the glyph contour set, builds a horizontal scan band per pair, and emits **one 4-vertex `TextFilledRegion` trapezoid per band per filled span**. A single curved glyph (e.g. 'O') produces dozens of trapezoids; a screenful of text produces hundreds of thousands. This is the structural source of the explosion. |
| Engine→protocol adapter | `crates/gui-protocol/src/lib.rs:3786` `board_text_graphics` | Iterates the `Vec<TextGeometryPrimitive>` from the engine and lowers each to `BoardGraphicPrimitive { primitive_kind: "polygon", source_object_uuid: text.uuid, ... }`. Note: `source_object_uuid` *does* preserve the text UUID per fragment, but every fragment is a separate protocol object that the renderer treats individually. Per-text grouping is structurally absent. |
| Protocol scene | `crates/gui-protocol/src/lib.rs:49` `BoardReviewSceneV1.board_graphics` | A flat `Vec<BoardGraphicPrimitive>`. Text fragments share the bucket with imported Edge.Cuts / mechanical authored polygons. |
| Protocol scene (semantic) | `crates/gui-protocol/src/lib.rs:51` `BoardReviewSceneV1.board_texts` | A parallel `Vec<BoardTextPrimitive>` carrying the semantic record (UUID, content, transform, font params). Used today only for hit testing (`board_text_hit_rect`) and dock chrome / property editing. **Decoupled from the rendered geometry.** |
| Renderer batching | `crates/gui-render/src/lib.rs:2766` `push_retained_board_graphic_batches` | Iterates `scene.board_graphics`, batches by render stage and layer_id. Per-fragment dispatch into `push_board_graphic_primitive_world`. |
| Renderer fill | `crates/gui-render/src/lib.rs:4168` `push_board_graphic_primitive_world` → `push_world_polygon_fill_contours` (`:6058`) | For 3-vertex / 4-vertex convex polygons, emits 1 triangle/quad. For everything else, falls back to `push_world_polygon_fill_scanline_contours` — another scanline pass at the renderer level. |

The architectural failure is right at the engine/protocol boundary:
the engine resolves text into a *flat list of fragments* with no
per-text grouping, the protocol encodes that flat list, and the
renderer iterates fragments. Even though `source_object_uuid` is
present on each fragment, no code path uses it to batch or to preserve
identity at draw time. **The protocol shape forces fragment-oriented
rendering.**

The good news: the engine semantic layer is clean (`BoardText`),
the existing outline machinery (`ttf-parser` + the in-house Bézier
flattener) is already deterministic and unit-tested, and the renderer
already has glyph-atlas infrastructure (`glyphon`) wired in for UI
chrome. The intermediate layers are what need to change.

## Six Focused Research Areas

### Area 1 — How serious CAD/vector/rendering systems avoid polygon confetti

The user's question 5 verbatim:

> How do serious CAD/vector/rendering systems avoid polygon-confetti
> representations for text?

The universal pattern, before per-system detail: **text is a first-
class object at every stage of the pipeline.** Systems converge on a
two- or three-tier model:

1. A **semantic text object** carries identity, content, font, and
   transform.
2. A **glyph layout** stage emits per-glyph positioned references
   (glyph_id + advance + transform) without ever materializing per-
   glyph polygons in the document model.
3. The **draw-time path** picks one of: (a) atlas blit (screen text),
   (b) cached vector mesh (zoomable engineering text), (c) on-GPU
   path tessellation (most modern, e.g. Vello).

Lowering text to anonymous polygons is the failure mode that every
mature system explicitly avoids, because it forfeits the cache
locality, the editing semantics, and the identity-preserving hit
test.

#### Skia (Chrome / Android / Flutter / many others)

Skia is the canonical reference. Its API exposes three text entry
points at progressively lower levels:

- `SkCanvas::drawString` / `SkCanvas::drawTextBlob` — semantic level,
  takes a string + font, hands off to the shaper.
- `SkCanvas::drawGlyphs` — post-shaping, takes glyph IDs + positions.
- `SkCanvas::drawPath` — fully lowered, takes a single `SkPath`.

The shaper (HarfBuzz when present) emits glyph IDs and positions, not
polygons. Skia's GPU backend rasterizes each glyph into a
**`GrAtlasManager`** texture atlas at the size first requested, keyed
by `(typeface_id, glyph_id, subpixel_position, color_emoji_flag)`.
Subsequent draws of the same glyph are GPU-instanced quads sampling
the atlas. For glyphs too large for the atlas (large display text),
Skia falls back to **path rendering** via `GrSmallPathRenderer` /
`GrTriangulator` / `GrTessellator` — i.e. it tessellates on demand
into triangles, but caches the path-to-triangle conversion in
`GrPathCache`. **Crucially, Skia never persists per-glyph polygons in
the document; the lowering is always render-time and always cached.**
Source: `src/gpu/ganesh/text/GrAtlasManager.h`, Skia documentation
"How Skia draws text".

For Datum's case, the relevant signal: Skia's API offers three
levels because three intents exist (semantic edit, glyph-positioned
draw, fully-lowered path). Datum's protocol type should mirror that —
keep the semantic record, expose the glyph-positioned form, and only
lower to triangles inside the renderer (or when exporting Gerber).

#### CoreText / CoreGraphics (macOS, iOS)

Apple's stack is the strongest example of *semantic identity flowing
all the way through.* The core type is **`CTLine`**, an immutable
shaped line that exposes:

- `CTLineGetStringRange` / `CTLineGetGlyphRuns` — semantic content.
- `CTLineDraw(line, ctx)` — atomic draw call; CoreText decides
  internally whether to use the system glyph cache or path rendering.
- `CTLineGetTypographicBounds` — exact metrics for layout and hit
  testing.

CoreText keeps per-glyph caches in a process-wide **glyph cache
server** (the WindowServer ATS daemon historically; now per-process
in modern macOS). The cache is keyed by `(font_descriptor, glyph_id,
size, subpixel_offset)`. A CTLine never decomposes into polygons at
the document level. Hit testing is `CTLineGetStringIndexForPosition`
— it returns the string index corresponding to a screen point, *not*
a polygon hit. This is the pattern Datum should mirror for board text
hit testing: hit a `BoardTextPrimitive` envelope, return the text
UUID, never a fragment ID.

(Authoritative info on CoreText's exact internal cache layout is not
publicly documented; the public API behavior, however, is well
documented in Apple's `CoreText.framework` headers.)

#### DirectWrite + Direct2D (Windows)

DirectWrite's central type is **`IDWriteTextLayout`** — a semantic
object holding a string, a default format, and per-range overrides
(font, color, weight, decorations). The draw call is:

```cpp
ID2D1RenderTarget::DrawTextLayout(origin, layout, brush, options);
```

Internally, Direct2D's text renderer translates this into a sequence
of glyph-run calls that hit the **DirectWrite font cache** (`IDWrite-
GdiInterop` glyph atlas). Glyphs are rasterized once into a shared
GPU atlas and instanced per draw. For glyphs above a size threshold
(roughly 100 px em), Direct2D switches to **path-based rendering**
through `ID2D1GeometrySink` and `ID2D1Mesh` — it tessellates the
glyph outline into a triangle mesh and caches the mesh.

The architectural lesson: **the choice between atlas and mesh is a
size-driven runtime decision, not a document-level commitment.** The
document holds semantic text; the renderer decides representation.
Datum's protocol type should match: carry semantic + glyph-positioned
form, let the renderer choose atlas vs mesh based on screen-space
glyph height.

#### HarfBuzz + Pango + Cairo (Linux GTK stack)

The three-layer split is most explicit here:

1. **HarfBuzz** shapes a Unicode string into a `hb_buffer_t` of
   `(glyph_id, x_advance, y_advance, x_offset, y_offset)`.
2. **Pango** wraps shaping in `PangoLayout` — semantic text with
   alignment, justification, multi-run formatting.
3. **Cairo** consumes glyph runs via `cairo_show_glyphs(cr, glyphs,
   num_glyphs)` — each glyph is a `cairo_glyph_t { index, x, y }`.

Cairo's font backend caches **`cairo_scaled_font_t`** per
`(font_face, font_matrix, ctm, options)`, and within each scaled
font caches glyph rasterizations on demand. The cache is essentially
a managed atlas; cairo will fall back to per-glyph path painting
(`cairo_show_text_glyphs` with `CAIRO_TEXT_CLUSTER_FLAG_BACKWARD`
metadata) if the consumer requested text-aware output (e.g. for
PDF/PS export, where glyph identity must survive in the file).

The relevant lesson for Datum: **even Cairo's vector PDF export
preserves glyph identity** in the output, with `cairo_show_text_-
glyphs` carrying cluster metadata back to the source string. Datum's
Gerber export must similarly emit text in a form that keeps glyph
identity alongside polygon fill (so a Gerber X3 attribute can carry
the text content for fab QC, even though the polygons remain the
manufacturable form).

#### wgpu_text / glyphon / cosmic-text (Rust ecosystem)

This stack is directly relevant — Datum's `gui-render` crate already
depends on `glyphon` for UI chrome (`crates/gui-render/Cargo.toml:14`,
`crates/gui-render/src/lib.rs:9`). The architecture:

- **`cosmic-text`** owns the text-shaping layer (HarfBuzz under the
  hood through `swash`'s shaping integration; layout via `cosmic-
  text::Buffer`). Output is `LayoutGlyph` per shaped glyph (glyph_id,
  position, font_id).
- **`glyphon`** owns the GPU text rendering layer. It maintains a
  `TextAtlas` (an `Etagere` binpacked texture atlas) of rasterized
  glyphs, a `SwashCache` of `swash`-rasterized glyph images, and a
  `TextRenderer` that emits one instanced quad per glyph at draw
  time.
- **`wgpu_text`** is a competing crate with similar shape, narrower
  scope (raster/atlas only, no shaping).

Why this matters for Datum: glyphon's `TextAtlas` + `TextRenderer`
gives Datum an off-the-shelf atlas implementation it has *already
linked*. The reason it cannot be used as-is for board text is
zoom invariance — glyphon rasterizes at the requested screen size,
which means a CAD zoom-in past the atlas size will pixelate. For UI
chrome (fixed screen size) it is correct. For board silkscreen
(arbitrary zoom) it is the wrong default. The right pattern is the
hybrid: glyph mesh cache (Datum-owned, world-space, zoom invariant)
for the common path, glyphon-style atlas as a future zoom-out
acceleration where the screen-space glyph height drops below ~12 px
and atlas blits become indistinguishable from mesh draws.

#### Vello / piet-gpu / Forma (Linebender GPU-path stack)

Vello (formerly piet-gpu) is the bleeding-edge alternative: **the
glyph outline goes to the GPU and a compute shader rasterizes it
per frame.** No atlas, no precomputed mesh. The stack:

- `vello::Scene` — semantic display list (paths, glyphs, transforms).
- `vello::Glyph` — `{ id, x, y }` with associated `Font` reference.
- `vello::Renderer` — submits compute dispatches; each frame resolves
  paths into GPU-side coverage masks via prefix-sum scan algorithms
  (the "Vello pipeline" paper, `linebender/vello` README).

Pros: zoom-invariant, no atlas resolution headache, no mesh cache to
invalidate. Cons: requires `wgpu` compute shaders (Datum already has
wgpu, so technically possible), high baseline GPU compute load (a
problem for laptop/integrated GPUs), and the pipeline is
**not yet pixel-deterministic** in the way Datum's golden-image
regression suite requires (open issue: vello relies on storage-buffer
atomics whose evaluation order can differ between drivers).

The forma engine (Google) has the same shape and the same caveats.

For Datum: **Vello/piet-gpu is the right long-term direction (Phase
3.x or M8) but not the right Phase 3.0 choice.** It would invalidate
the visual-regression authority and force an immediate cross-driver
determinism investigation. The mesh cache + per-text batching path is
a strict subset of what Vello does internally, so a future Vello
migration is a backend swap, not a redesign.

#### AutoCAD MTEXT

AutoCAD's multi-line text object (`MTEXT`) is the design-software
analog of Datum's `BoardText`. The internal representation is
**fully semantic at the document level**: content, justification,
font (SHX or TTF), height, width-factor, oblique angle, line spacing,
column layout. Tessellation is on-demand and cached per session.

The user-visible failure mode of treating MTEXT as polygons is
catastrophic — once you `EXPLODE` an MTEXT into individual line
segments and arcs (`SHX` font path), you can never edit the text
again. AutoCAD's UI calls this out explicitly. The same is true for
Datum: lowering `BoardText` to `BoardGraphicPrimitive` fragments at
the protocol level is the architectural equivalent of `EXPLODE`,
applied silently at every render.

For SHX (stroke) fonts, AutoCAD ships a per-session `SHX cache` of
line segments per glyph. For TTF fonts, AutoCAD uses Windows GDI's
glyph cache plus its own polygonization cache for printed/3D output.
The distinction Datum needs to mirror: **representation per intent,
identity per object.**

#### SolidWorks / Inventor / Creo

Mechanical CAD systems treat annotation text as a **note feature**
(SolidWorks `Note`, Inventor `Text`, Creo `note`). The note holds
content, leader information, anchor, and font reference. Display in
the graphics window goes through the host platform's text renderer
(usually GDI+ on Windows, Cairo/X11 on Linux clients). When the note
is exported (DXF, IGES, PDF), the text is *re-shaped at export time*
into polylines or kept as a font reference depending on the export
target's capability.

Identity is preserved at every stage. The polygonization, when it
happens, happens at export time and is not persisted in the model.

#### KiCad's `KIGFX::PCB_PAINTER` with `OUTLINE_FONT`

This is the closest peer to Datum and the most important comparison.
KiCad's PCB painter (`pcbnew/pcb_painter.cpp`) handles board text
through a small dispatch:

```cpp
void PCB_PAINTER::draw( const PCB_TEXT* aText, int aLayer ) {
    KIFONT::FONT* font = aText->getFont();
    font->Draw( m_gal, aText->GetShownText(), aText->GetTextPos(), attrs );
}
```

The interesting fork is inside `KIFONT::OUTLINE_FONT::Draw` (kicad
source `libs/kifont/outline_font.cpp`):

1. Shape the string with HarfBuzz to get glyph IDs.
2. For each glyph, pull cached outline contours from
   `KIFONT::OUTLINE_FONT::m_glyphCache` (a per-font glyph contour
   cache keyed by glyph_id; the contours are flattened Bézier
   polylines, the same shape Datum's `flatten_glyph_from_face`
   produces).
3. For each contour, call `m_gal->DrawPolygon( pts )`. The GAL
   (Graphics Abstraction Layer) is responsible for tessellation.

**KiCad's GAL backends do exactly the same thing Datum is doing
today.** The OpenGL GAL (`gal/opengl/opengl_gal.cpp`) calls into
`tesselator.cpp`'s `TESSELLATOR::Tesselate`, which is GLU's tess
algorithm wrapped, and produces **per-frame triangle output**. The
Cairo GAL fills polygons through Cairo. **There is no per-glyph mesh
cache in KiCad — the tessellation happens every frame.**

This is why KiCad PCB editor performance with TTF text on dense
silkscreen layers is a known sore point (KiCad bug tracker:
`#11389 "outline font silkscreen rendering very slow"`,
`#13728 "OUTLINE_FONT performance with many text objects"`, both
open as of repo memory). KiCad has the **glyph contour cache** but
not the **glyph mesh cache** and not the **per-text batched draw**.

The architectural opportunity: **Datum can do better than KiCad on
the same problem by adding the glyph mesh cache + per-text batching
that KiCad has not yet built.** This is not a hypothesis — it is the
explicit gap in the closest peer codebase.

#### Universal pattern across all systems surveyed

| Property | Skia | CoreText | DirectWrite | Pango/Cairo | glyphon | Vello | AutoCAD | Mechanical CAD | KiCad GAL |
|----------|------|----------|-------------|-------------|---------|-------|---------|----------------|-----------|
| Semantic text object survives end-to-end | yes | yes | yes | yes | yes | yes | yes | yes | yes |
| Per-glyph cache (atlas or contour) | yes | yes | yes | yes | yes | n/a (compute) | yes | yes (via host) | contour only |
| Per-glyph mesh cache | yes | implementation-private | yes | implementation-private | no (atlas only) | n/a | yes | yes | **no** |
| Per-text batching | yes | yes | yes | yes | yes | yes | yes | yes | partial |
| Object-level hit test | yes | yes | yes | yes | yes | yes | yes | yes | yes |
| Document holds polygons (anti-pattern) | no | no | no | no | no | no | no | no | no |

Every mature system avoids polygon-at-the-document-level. Datum is
the outlier today.

### Area 2 — Glyph atlas vs vector path vs hybrid (GPU side)

The user's central performance question. Five candidates:

#### Option A — Glyph atlas + instanced quads

How: rasterize each glyph once at a representative pixel size (e.g.
64 px em) into a packed texture atlas; at draw time, every glyph
instance is one quad sampling the atlas at the right offset.

Pros (specific):
- O(1) per glyph at draw time, regardless of glyph complexity.
- Already implemented in `glyphon`, already linked in `gui-render`.
- Tiny GPU memory cost: a typical atlas of all printable ASCII for
  one font is ~512 KB at 64 px em.
- Excellent batching: every text instance hits the same atlas
  texture, draws collapse to one or two quad batches per scene.

Cons (specific):
- **Zoom-in past the atlas resolution pixelates.** A board zoomed
  from "fit board" to "single 0402 silkscreen letter" can span 100×
  scale change. Atlas-only fails this test.
- Re-rasterizing at multiple zoom levels reintroduces a multi-atlas
  cache and the cost-per-zoom-level problem.
- Subpixel positioning (needed for crisp screen text) requires either
  4× atlas storage (one per quarter-pixel offset) or filtering, both
  of which break golden-image determinism across drivers.
- World-space coordinate goldens are brittle: the atlas pixel grid
  is *screen-space*, not world-space, so any change in viewport
  scale shifts where pixels land in world units.
- Hinting / weight changes invalidate the atlas wholesale.

Verdict for Datum: **wrong as the only path.** Zoom-in pixelation is
disqualifying for a CAD tool. Right as a deferred Phase 3.2 path for
zoom-out screen-text acceleration where world-space accuracy
matters less than throughput.

#### Option B — CPU-tessellated mesh per glyph, cached

How: for each glyph, run the outline through `lyon`'s `FillTessella-
tor` with `FillRule::NonZero`, store the resulting `(Vec<Vertex>,
Vec<u16>)` keyed by `(font_id, glyph_id, tolerance_class)`. At draw
time, instance the cached mesh per text-character position.

Pros:
- Zoom-invariant: a glyph mesh is in font em units, scaled at draw
  time by the per-text transform; pixelation is impossible.
- One source of truth: same mesh feeds the renderer and (after a
  trivial conversion) the manufacturing export pipeline.
- Cache locality: a typical English-language fixture uses ~70
  distinct glyphs across all text on the board; the cache fits in
  ~50–200 KB total.
- Determinism: `lyon`'s `FillTessellator` is deterministic on a fixed
  host architecture; the same input contour produces the same
  triangle list bit-for-bit.
- Plays well with existing wgpu pipeline: one extra vertex buffer
  binding per per-text batch, no new shader.

Cons:
- More vertices than atlas (one mesh has 10–80 triangles for a
  typical glyph; an atlas instance is 2 triangles).
- Tessellation is not free: cold cache builds for a 70-glyph font
  take ~5–15 ms. (Subsequent frames pay nothing.)
- Self-intersecting Bézier paths (decorative fonts) need pre-cleanup
  (cavalier_contours / i_overlay) or `lyon` will reject them. Phase 2
  already accounted for this.

Verdict for Datum: **the right default.** Zoom invariance and
determinism are non-negotiable; glyph counts on a typical board (1k–
10k characters) keep the per-frame triangle budget within 100k–500k
triangles, well within wgpu's comfortable throughput on the
software-Mesa CI authority.

#### Option C — Vector path on GPU (Vello / piet-gpu)

How: skip CPU tessellation entirely; submit compute dispatches that
rasterize glyph paths directly into coverage masks per frame.

Pros:
- True zoom invariance with no cache.
- Subpixel-perfect anti-aliasing.
- Becomes more attractive as compute hardware improves.

Cons:
- **Not pixel-deterministic across drivers today.** Cross-driver and
  cross-architecture pixel agreement is an open Vello issue. Datum's
  visual-regression authority requires deterministic goldens; this
  rules Vello out for Phase 3.0.
- Requires a fully separate render pipeline, parallel to the existing
  quad pipeline.
- Compute load is 5–10× higher than mesh instancing on integrated
  GPUs.
- API churn: Vello is at v0.2; the API is still moving.

Verdict for Datum: **wrong for Phase 3.0; possibly right for M8 as a
backend swap.** The mesh-cache architecture is a strict subset of
Vello internals, so a future migration is non-destructive.

#### Option D — Signed Distance Field (SDF) atlas

How: rasterize each glyph as a distance field (each pixel encodes
distance to the nearest glyph edge) at a fixed resolution; sample
the field at draw time and threshold for coverage. Zoom-invariant up
to ~16× scale before edges visibly degrade.

Pros:
- Zoom-invariant in the practical range.
- Atlas-style throughput.
- One-time GPU rasterization cost per glyph.

Cons:
- **Thin strokes and sharp corners artifact.** PCB silkscreen text is
  *both* (0.15 mm stroke at 1 mm height = 1:6 stroke ratio, and many
  fonts have sharp corners on serifs and angled strokes). SDF
  blunts both.
- Multi-channel SDF (MSDF) fixes corners but doubles atlas storage
  and compute.
- Determinism across GPU vendors is poor (driver-specific
  derivative-instruction implementations).

Verdict for Datum: **wrong.** PCB silkscreen is exactly the worst
case for SDF.

#### Option E — Hybrid: mesh cache primary, atlas for screen-space acceleration

How: glyph mesh cache (Option B) is the always-on default. A future
phase adds a screen-space atlas layer that activates when zoom-out
makes the mesh path overdraw-bound (typical threshold: glyph screen
height < ~12 px). The atlas hands off to mesh as soon as zoom passes
the threshold the other direction.

Pros:
- Best of both worlds for the actual usage envelope.
- Migration: ship Option B in Phase 3.0; layer atlas on top in 3.2
  without invalidating the document model.

Cons:
- More code than Option B alone.
- Hand-off rules need careful golden testing to prevent visible
  popping at the threshold.

Verdict for Datum: **the recommended trajectory.** Phase 3.0 ships
mesh-only (Option B). Phase 3.2 adds atlas acceleration if profiling
warrants it.

#### Final recommendation — GPU representation

Phase 3.0: **Option B (cached glyph mesh + per-text instancing)** as
the only path.

Phase 3.2 (if needed): add the atlas as zoom-out acceleration; keep
Option B as world-space ground truth.

Reject for Phase 3.x: SDF (artifacts unacceptable on PCB strokes),
Vello (determinism not yet).

### Area 3 — Where tessellation happens (the user's question 4)

Five candidate stages:

| Stage | Pros | Cons | Verdict |
|-------|------|------|---------|
| **Engine** | Single source of truth (renderer + Gerber export consume the same mesh). Determinism contract already lives here. Cache amortizes across boards/fixtures. Survives `engine`-only consumers (CLI, MCP, headless tests). | Engine pays the cost even when consumer doesn't need triangles. Mitigated by lazy materialization. | **Recommended** |
| **gui-protocol adapter** | Engine stays pure. | Re-tessellates on every protocol message. Cache lives in adapter, invisible to other consumers. Duplicates engine work. | Reject |
| **gui-render retained scene** | Renderer optimizes for its own batching layout. | Every renderer consumer (current GUI, future review surface, future native renderer) re-implements. Cache invalidation tangled with renderer state. | Reject |
| **GPU/render backend** | Zoom-invariant. | Vello immaturity (Area 2). Driver determinism. | Defer to M8 |
| **Export pipeline** | Manufacturing output already needs polygon form. | Already required regardless; not a real choice. Engine path should also feed export. | Co-resident with engine |

The recommendation: **engine owns tessellation, with a content-
addressed glyph mesh cache.** The protocol adapter selects from the
cache and packages per-text batches. The renderer consumes pre-
tessellated meshes and arranges them into per-layer quad batches. The
export pipeline reads the same mesh cache (or, for Gerber-specific
needs, asks the engine for filled polygons via a separate call that
shares the cache key).

#### Why engine, specifically

1. **Determinism boundary.** The engine already enforces the
   x86_64-linux determinism contract for routing, DRC, and import.
   Putting tessellation in the engine keeps determinism in one place.
2. **Cache lifetime matches engine session.** Glyph meshes are
   font-content; they live as long as the font is loaded, which is
   an engine concern.
3. **Multi-consumer reuse.** The engine is the shared substrate for
   CLI batch fixtures, MCP queries, the GUI renderer, the future
   visual-regression harness, and Gerber/ODB++ export. All of them
   benefit from cached mesh.
4. **Test parity.** Engine-level unit tests can assert mesh
   determinism without a GPU. The renderer can stay focused on
   pipeline behavior.

#### Lazy materialization

The engine should materialize meshes on demand, not eagerly per
operation:

```rust
pub fn ensure_text_mesh(&mut self, text_uuid: Uuid) -> &TextMeshHandle;
```

Calling `ensure_text_mesh` is idempotent. Operations that mutate text
(`SetBoardTextContent`, `SetBoardTextHeight`, etc.) invalidate the
mesh handle for that UUID; the next consumer that calls
`ensure_text_mesh` triggers the rebuild. Glyph-level cache entries
survive across text mutations — only the per-text mesh assembly is
rebuilt, and it reuses cached glyph meshes by key.

### Area 4 — Caching model (the user's question 7)

Five-tier hierarchy. Concrete cache keys and invalidation rules:

#### Tier 1: Glyph contour cache (engine-owned)

```rust
pub struct GlyphContourKey {
    pub font_id: FontId,
    pub glyph_id: GlyphId,
    pub tolerance_class: ToleranceClass,
    pub bold_axis_value_micro: Option<u32>,
    pub italic_axis_value_micro: Option<u32>,
    pub optical_size_axis_value_micro: Option<u32>,
}

pub struct GlyphContours {
    pub contours: Vec<Vec<Point>>,    // em-unit Points (i64 nm)
    pub bbox_em: BBox,
    pub advance_em: i64,
}
```

**Invalidates on**: font reload, font axis change. **Survives**:
text mutation, transform change, layer change, layout change.

`tolerance_class` is a coarsely-bucketed text size class (e.g.
"small" ≤ 1.5 mm height, "medium" ≤ 6 mm, "large" > 6 mm). This
prevents one-pixel cache thrash from per-text continuous size sliders
while still keeping fine-text and hero-text quality separate.

Storage: ~200 bytes per glyph (typical glyph: 4 contours × 12 points
× 16 bytes). A 95-character ASCII font with three tolerance classes:
~57 KB per font.

#### Tier 2: Glyph tessellated mesh cache (engine-owned)

```rust
pub struct GlyphMeshKey {
    pub contour_key: GlyphContourKey,  // same key, derived
    pub fill_rule: FillRule,           // always NonZero per Phase 1
}

pub struct GlyphMesh {
    pub vertices: Vec<MeshVertex>,    // (x_em_nm, y_em_nm) i64
    pub indices: Vec<u32>,
    pub bbox_em: BBox,
    pub triangle_count: u32,
}
```

**Invalidates on**: font reload, font axis change, fill_rule change.
**Survives**: text mutation, transform change.

Storage: ~600 bytes per glyph (10–40 triangles × ~30 bytes per
triangle in indexed form). 95-character ASCII font: ~57 KB.

#### Tier 3: Text layout cache (engine-owned)

```rust
pub struct TextLayoutKey {
    pub text_uuid: Uuid,
    pub content_hash: u64,            // xxhash3 of UTF-8 bytes
    pub font_id: FontId,
    pub style_id: StyleId,
    pub height_class: HeightClass,
    pub stroke_width_class: StrokeWidthClass,
    pub h_align: TextHAlign,
    pub v_align: TextVAlign,
    pub mirrored: bool,
    pub keep_upright: bool,
    pub line_spacing_ratio_class: LineSpacingClass,
    pub italic: bool,
    pub bold: bool,
    pub style_class_hash: Option<u64>,
}

pub struct TextLayout {
    pub positioned_glyphs: Vec<PositionedGlyph>,
    pub block_bbox_em: BBox,
}

pub struct PositionedGlyph {
    pub glyph_id: GlyphId,
    pub origin_em: PointEm,           // unscaled; transform applied later
    pub mesh_handle: GlyphMeshHandle,  // strong reference into Tier 2
}
```

**Invalidates on**: any of the keyed semantic-text attributes
changing. Identical content + identical font + identical attrs hits
the cache regardless of position/rotation (those go in Tier 4).

#### Tier 4: Text geometry batch (engine-owned, per text object)

```rust
pub struct TextGeometryBatch {
    pub text_uuid: Uuid,
    pub layer: LayerId,
    pub world_transform_nm: Affine,   // i64-fixed-point affine
    pub layout_handle: TextLayoutHandle,
}
```

**Invalidates on**: world transform change (text moved/rotated),
layer change, or layout invalidation. The transform-only invalidation
case is the cheapest — it does not touch glyph meshes or layout, only
re-stamps the affine.

#### Tier 5: Renderer GPU buffer cache (renderer-owned)

```rust
pub struct GpuTextMeshKey {
    pub mesh_handle: GlyphMeshHandle,  // engine-side handle, opaque to renderer
    pub epoch: u64,                    // bumps when engine evicts the handle
}

pub struct GpuTextMesh {
    pub vertex_buffer: wgpu::Buffer,
    pub index_buffer: wgpu::Buffer,
    pub index_count: u32,
}
```

**Invalidates on**: engine epoch bump (rare, only on font reload),
GPU device loss.

#### Cache invalidation flowchart

```
SetBoardTextContent (UUID)        → Invalidates Tier 3 + Tier 4 for UUID.
                                    Tier 1 + Tier 2 hot.
SetBoardTextHeight (UUID)         → Invalidates Tier 3 + Tier 4 for UUID
                                    only if height crosses a HeightClass
                                    boundary; transform-only otherwise.
SetBoardTextRotation (UUID)       → Invalidates Tier 4 only.
SetBoardTextPosition (UUID)       → Invalidates Tier 4 only.
SetBoardTextLayer (UUID)          → Invalidates Tier 4 only.
SetBoardTextFamily (UUID)         → Invalidates Tier 3 + Tier 4 for UUID.
                                    Tier 2 hot for other UUIDs using same font.
LoadFont (font_id)                → Invalidates Tier 1, 2 for font_id;
                                    Tier 3, 4 cascade for any UUID using it.
```

The user's invariant — "invalidate only when semantic text attributes
change" — is satisfied: Tiers 1 and 2 are font-content, immune to
text mutation; Tier 3 only invalidates on attribute change; Tier 4
isolates the cheap transform-only path.

### Area 5 — Protocol shape (the user's question 2)

Three concrete options.

#### Option A — `BoardTextGeometry` per text object, opaque payload

```rust
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq)]
pub struct BoardTextGeometry {
    pub object_id: String,
    pub object_kind: String,         // "board_text_geometry"
    pub text_uuid: String,
    pub layer_id: String,
    pub world_transform_ppm: Affine2D,
    pub payload: BoardTextGeometryPayload,
}

#[derive(Debug, Clone, Serialize, Deserialize, PartialEq)]
#[serde(tag = "kind", rename_all = "snake_case")]
pub enum BoardTextGeometryPayload {
    Outline { contours: Vec<Vec<PointNm>>, holes: Vec<Vec<Vec<PointNm>>> },
    Mesh    { vertices: Vec<MeshVertex>, indices: Vec<u32> },
    Atlas   { glyphs: Vec<AtlasGlyphRef> },
}
```

Pros: One protocol message per text object, regardless of glyph count.
Renderer chooses representation per render call.

Cons: The `Outline` case can still be large for hero text. The renderer
has no per-glyph instancing handle — it sees one giant mesh per text
object, not many cached glyph meshes.

#### Option B — `BoardTextBatch` per layer/draw-pass

```rust
pub struct BoardTextBatch {
    pub layer_id: String,
    pub texts: Vec<BoardTextBatchEntry>,
}

pub struct BoardTextBatchEntry {
    pub text_uuid: String,
    pub world_transform_ppm: Affine2D,
    pub glyphs: Vec<AtlasGlyphRef>,
}
```

Pros: Maximum batching, single message carries an entire layer's text.

Cons: Per-text identity sits inside batch entries — at risk if
batching consolidates entries. Hit testing on the renderer side has
to walk into the batch. Snapshot review surfaces (the future review
proposal viewer) lose the natural "one text = one record"
correspondence.

#### Option C — Hybrid: `BoardTextGeometry` per text object, payload is a glyph-instance batch

```rust
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq)]
pub struct BoardTextGeometry {
    pub object_id: String,
    pub object_kind: String,         // "board_text_geometry"
    pub text_uuid: String,
    pub layer_id: String,
    pub world_transform_ppm: Affine2D,
    pub glyphs: Vec<TextGlyphInstance>,
    pub block_bbox_nm: RectNm,        // for selection / hit envelope
}

#[derive(Debug, Clone, Serialize, Deserialize, PartialEq)]
pub struct TextGlyphInstance {
    pub glyph_handle: GlyphMeshHandle,    // opaque, engine-issued, stable per session
    pub origin_em_x: i64,                 // glyph origin in unscaled em units
    pub origin_em_y: i64,
}

#[derive(Debug, Clone, Copy, Serialize, Deserialize, PartialEq, Eq, Hash)]
pub struct GlyphMeshHandle {
    pub font_id: u32,
    pub glyph_id: u32,
    pub tolerance_class: u8,
    pub epoch: u32,                       // bumps on font reload
}
```

Pros:
- One message per text object — identity preserved.
- Renderer can fetch the mesh from a renderer-side `GlyphMeshHandle →
  GpuTextMesh` table; the heavy mesh data is *not* in the protocol
  message, only the handle.
- Per-text affine + per-glyph origin gives the renderer everything it
  needs to instance.
- Batching happens inside the renderer per layer / per render stage,
  same shape as `push_retained_board_graphic_batches` today, but
  driven by `BoardTextGeometry` instead of `BoardGraphicPrimitive`.
- The mesh handle is opaque to the protocol (no float content),
  preserving golden-image determinism over JSON serialization.

Cons:
- Requires a renderer-side handle resolver — the renderer needs an
  out-of-band channel to receive the actual mesh data when a handle
  is first seen.
- Renderer must handle handle-cache misses (engine may have evicted).

#### How the handle channel works (Option C detail)

The protocol gains a parallel `Vec<GlyphMeshAsset>` carried at scene
construction time:

```rust
pub struct GlyphMeshAsset {
    pub handle: GlyphMeshHandle,
    pub vertices: Vec<MeshVertex>,    // em-unit
    pub indices: Vec<u32>,
}

pub struct BoardReviewSceneV1 {
    // ... existing fields ...
    pub board_texts: Vec<BoardTextPrimitive>,         // semantic, kept
    pub board_text_geometries: Vec<BoardTextGeometry>,// new
    pub glyph_mesh_assets: Vec<GlyphMeshAsset>,       // new
    // ... existing fields ...
}
```

The renderer, on receiving a scene, builds its `GlyphMeshHandle →
GpuTextMesh` table from `glyph_mesh_assets`, then iterates
`board_text_geometries` and instances. Mesh assets are only included
if the scene actually contains text using them — the cost is
proportional to distinct glyphs in the scene, not total text count.

**For the `text-density-repro` fixture**: dozens of distinct glyphs
across thousands of text instances. Asset count: ~70. Instance count:
thousands. Compared to today's 347,154 polygon fragments, this is a
~5,000× reduction in protocol message count.

#### Recommendation: Option C

Option A loses per-glyph instancing and inflates payload for hero
text. Option B fragments identity across batches. Option C preserves
identity per text, batches at the glyph level via cached handles, and
keeps the protocol message lean.

The semantic record `BoardTextPrimitive` stays as-is — it remains
the source of identity for hit testing, selection, and property
editing. `BoardTextGeometry` is the **render-only** companion. The
renderer never reads `BoardTextPrimitive` to build geometry, and the
hit-testing layer never reads `BoardTextGeometry` to find text.

gating dependency.

## Answers to the User's 10 Questions

> 1. What should the boundary be between: semantic
>    `BoardTextPrimitive` / generated text geometry / renderable
>    mesh-or-primitive data / manufacturing-export geometry?

`BoardText` (engine semantic) → `BoardTextPrimitive` (protocol
semantic) carries identity, content, transform, and font params.
`TextGeometryBatch` (engine derived, content-addressed cache) carries
positioned glyphs + cached mesh handles. `BoardTextGeometry`
(protocol render-only) wraps the batch with a stable per-text
identifier and per-frame world transform. `GlyphMeshAsset` (protocol
asset table) carries the actual mesh data once per distinct glyph
per scene. Manufacturing-export geometry comes from a separate engine
call (`text_filled_polygons_for_export(text_uuid) -> Vec<Polygon>`)
that shares the same Tier 1/2 caches; the export path emits filled
polygons (not triangle meshes) because Gerber consumes polygons. See
Areas 4 and 5.

> 2. Should Datum introduce a dedicated protocol type such as
>    `BoardTextGeometryPrimitive`, `BoardTextMesh`,
>    `TextGeometryBatch`, or another grouped representation?

Yes. The recommended type is `BoardTextGeometry` per text object
(Option C in Area 5), carrying per-text affine + ordered
`Vec<TextGlyphInstance>`, plus a parallel `Vec<GlyphMeshAsset>` at
scene level for the actual mesh payload. The semantic
`BoardTextPrimitive` continues to exist for hit testing and editing.
See Area 5.

> 3. Should outline text geometry be represented as filled polygons
>    with holes, tessellated triangles, compact path contours,
>    GPU-side path/curve data, cached mesh objects, or multiple
>    representations depending on intent?

Multiple representations, organized by intent:
- **Engine internal Tier 1**: flattened em-unit Bézier contours
  (already exists today).
- **Engine internal Tier 2**: tessellated triangle mesh per glyph,
  cached.
- **Protocol render path**: per-text glyph instances + mesh handles
  (Option C).
- **Protocol export path** (Gerber, ODB++): filled polygons with
  holes, generated from Tier 1 contours + a unioning step
  (`i_overlay` per Phase 2).
- **GUI atlas path** (Phase 3.2): screen-space rasterized glyph atlas,
  derived from Tier 1 contours.
None of these representations replaces another; they coexist, share
cache infrastructure, and are selected per consumer.
See Areas 2, 3, and 4.

> 4. Where should tessellation happen — engine, gui-protocol adapter,
>    gui-render retained scene, GPU/render backend, or export
>    pipeline?

Engine, with a content-addressed cache. The protocol adapter and the
renderer are pre-tessellation consumers. Reasoning: the engine is the
authoritative determinism boundary, the cache amortizes across all
downstream consumers, and the engine is the only place the
manufacturing-export pipeline can also share the cache. GPU-side
tessellation (Vello) is the right long-term direction but not in
Phase 3.0 due to cross-driver determinism gaps. See Area 3.

> 5. How do serious CAD/vector/rendering systems avoid polygon-confetti
>    representations for text?

By keeping the semantic text object as the canonical document
representation and lowering to polygons / meshes / atlas blits only
at draw time, with caches at every stage. Skia, CoreText,
DirectWrite, Pango+Cairo, AutoCAD, mechanical CAD, and even KiCad's
GAL all do this — *KiCad's gap is that it tessellates per frame
instead of caching the mesh, which is exactly the gap Datum can close
to do better than the closest peer.* See Area 1.

> 6. What is the right ownership model? (Engine owns semantic layout
>    and deterministic text geometry; GUI protocol transports grouped
>    geometry; GUI renderer consumes grouped/batched render geometry;
>    renderer should not infer semantic text from arbitrary board
>    graphics.)

The user's framing is correct in full and is what this report
recommends. To formalize: engine owns Tiers 1–4 (Area 4); protocol
transports `BoardTextPrimitive` (semantic) and `BoardTextGeometry` +
`GlyphMeshAsset` (rendered form); renderer owns Tier 5 (GPU buffer
cache); renderer never reads `BoardGraphicPrimitive` to infer text.
See Area 5.

> 7. What is the right caching model? (Cache by text UUID + content +
>    font + size + stroke/fill params + transform; cache generated
>    contours; cache tessellated mesh; cache GPU buffers; invalidate
>    only when semantic text attributes change.)

The user's framing is correct. The detailed five-tier hierarchy
formalizes it: Tier 1 (font-content glyph contours), Tier 2
(font-content glyph meshes), Tier 3 (per-text-UUID layout), Tier 4
(per-text-UUID transform-applied batch), Tier 5 (renderer GPU
buffers). Tiers 1–2 invalidate on font reload only; Tier 3 on
semantic-attribute change; Tier 4 on transform change; Tier 5 on
engine epoch bump. The transform-only invalidation case keeps move/
rotate operations cheap. See Area 4.

> 8. How should hit testing work? (Current direction is correct: one
>    hit target per text object. Confirm whether hit testing should
>    use semantic text bounds, outline bounds, or selectable
>    envelope.)

Confirmed: one hit target per text object, keyed by text UUID. The
hit envelope is the **selectable envelope** — `block_bbox_nm` from
the engine layout, padded to the user's selection-comfort margin (the
existing `board_text_hit_rect` at `crates/gui-render/src/lib.rs:3403`
already does this with a hard-coded `(height * 0.25).max(250_000.0)`
padding; promote that constant to a per-render-intent setting). The
outline bounds (the actual glyph-fill silhouette) is wrong for hit
testing because counter-spaces (the inside of an 'O') would miss.
The semantic text bounds (line metrics) is wrong because it ignores
descenders. See Area 5.

> 9. How should this interact with manufacturing outputs? (Beautiful
>    outline fonts should be allowed for manufacturing text;
>    fabrication constraints should be validation/DRC concerns, not
>    renderer degradation; manufacturing export may require polygon/
>    Gerber-safe output, but GUI representation should not be forced
>    into the same fragmented form.)

The user's framing is correct. The architecture supports it:
- Engine Tier 1 (glyph contours) feeds both the GUI mesh path *and*
  the manufacturing polygon path. Same source of truth, different
  consumers.
- The Gerber export pipeline calls `text_filled_polygons_for_export`
  on the engine, which derives Gerber-safe polygons from the same
  cached contours via `i_overlay` boolean union (per Phase 2's stack
  recommendation). This output is *only* for the export pipeline; it
  never surfaces in the GUI scene.
- DRC validates fabricability (minimum stroke width, minimum text
  height per Phase 2's clamp policy) at validation time, not at
  render time. Renderer never degrades for fabrication concerns.
- A future "render manufacturing polygons preview" view in the GUI
  is supported as a separate scene primitive layer, not by changing
  the canonical text rendering path.
See Areas 3 and 4.

> 10. What libraries or algorithms are appropriate — lyon
>     tessellation, kurbo path representation, ttf-parser outline
>     extraction, custom deterministic tessellator, CPU mesh cache,
>     GPU path rendering alternatives?

Recommended Phase 3.0 stack:
- `ttf-parser` 0.21+ — outline extraction (already in use).
- In-house Bézier flattener (`crates/engine/src/text/outline.rs`,
  already deterministic, already unit-tested) — keep as-is for Tier
  1.
- `lyon_tessellation` 1.0+ — fill triangulation for Tier 2. Apache-
  2.0 / MIT licensed. Deterministic on a fixed host architecture.
- `i_overlay` 1.0+ — boolean union for the Gerber-export polygon
  path (per Phase 2). Apache-2.0 / MIT.
- `kurbo` 0.11+ — only if Tier 1 needs to be replaced; not required
  for Phase 3.0 (in-house flattener is fine). Apache-2.0 / MIT.
- `glyphon` 0.10 (already linked) — defer to Phase 3.2 for atlas.
- Reject for Phase 3.0: Vello (cross-driver determinism not
  established), `swash` (rasterization, atlas-only).
See Area 2 and Phase 2 research.

### Cache keys (tabulated)

| Tier | Key | Stored value | Invalidates on |
|------|-----|--------------|----------------|
| 1 | `GlyphContourKey` (font_id, glyph_id, tolerance_class, font axes) | em-unit flattened Bézier contours | font reload, font axis change |
| 2 | `GlyphMeshKey` (= contour_key + fill_rule) | em-unit triangle mesh | font reload, font axis change |
| 3 | `TextLayoutKey` (text_uuid, content_hash, font, style, classes, alignment, mirror, keep_upright, italic, bold) | positioned glyphs + bbox | semantic-attribute change for text |
| 4 | `(text_uuid, world_transform_hash)` | engine-side `TextGeometryBatch` | transform change for text, or Tier 3 invalidation |
| 5 | `GlyphMeshHandle` (engine-issued opaque) | wgpu vertex+index buffers | engine epoch bump, GPU device loss |

### Renderer responsibilities

1. Maintain a per-frame `HashMap<GlyphMeshHandle, GpuTextMesh>` from
   the scene's `glyph_mesh_assets`. Reuse buffers across frames where
   the handle is the same and the epoch matches.
2. Iterate `scene.board_text_geometries` grouped by `layer_id` and
   render stage (the existing `RenderStage` enum: `BottomMask`,
   `TopMask`, ..., `Mechanical`, `Edge`, plus future copper-text
   layers).
3. For each `BoardTextGeometry`, push one GPU draw indexed call per
   glyph instance (or, with future instancing, one indirect draw per
   text). The per-glyph draws are tiny (10–80 triangles each) and
   instance-shareable.
4. Apply `world_transform_nm` × per-glyph `origin_em_nm` × cached
   mesh in the vertex shader. The renderer's existing world-space
   coordinate system is preserved — text vertices are world-space nm
   like every other primitive.
5. Hit testing remains unchanged: walk `scene.board_texts` (the
   semantic record) and call `board_text_hit_rect` per text. The
   geometry stream is render-only.

### Engine responsibilities

1. Maintain Tiers 1–4 as in-process LRU caches (capacity bounded by
   bytes; no time-based eviction, no IO).
2. Expose `ensure_text_mesh(text_uuid) -> TextGeometryBatchHandle`
   for the protocol adapter.
3. Expose `text_filled_polygons_for_export(text_uuid) -> Result<
   Vec<Polygon>, ExportError>` for the manufacturing path.
4. Bump `epoch` on font reload; this cascades through Tier 1–4
   caches.
5. Preserve all existing engine APIs unchanged (semantic operations,
   query surface).

### Protocol responsibilities

1. Carry `BoardTextPrimitive` (semantic, identity, hit testing) and
   `BoardTextGeometry` (render-only, per text). Both keyed by
   `text_uuid`.
2. Carry `glyph_mesh_assets` once per scene at the
   `BoardReviewSceneV1` level. Asset count = distinct (font_id,
   glyph_id, tolerance_class) triples in the scene.
3. Stay deterministic: all numeric values are integer nm or content-
   addressed handles. Float content (the affine transform's rotation
   matrix) is the only float in the protocol; it derives from
   `rotation_degrees: i32` and is reproducible.
4. Be backward-compatible at the JSON level: all new fields
   `#[serde(default)]`, old fixtures continue to round-trip.

The Phase 3.0 numbers target a ~10–30× speedup on retained-scene

path remains the canonical world-space form.

### Known risks

1. **`lyon` tessellator non-determinism across host architectures.**
   Float Bézier flattening (Phase 2's named risk) cascades into
   triangle mesh determinism in Phase 3. Mitigation: x86_64-linux +
   Mesa is the golden authority; aarch64 is checked-against-tolerance;
   tolerance_class bucketing prevents cache thrash.
2. **Cross-driver float drift in the world-transform matrix multiply.**
   The vertex shader multiplies em-unit positions by float affine.
   Mitigation: integer translation component (`m02_nm`, `m12_nm`),
   float rotation/scale only. Existing world-space coordinate
   determinism is preserved.
3. **Cache memory pressure with thousands of text objects.** Tier 3
   stores a `Vec<PositionedGlyph>` per text UUID; for 10,000 text
   objects on a large board, this is ~3–5 MB. Acceptable. Tier 1 and
   Tier 2 are font-content; capped by font character set
   (~500 KB total for the 5-font bundle from Phase 2).
4. **Renderer-backend portability if GPU-side path rendering ever
   lands.** Vello migration would require replacing the
   `GlyphMeshAsset` → GPU buffer path with Vello's path-stream
   submission. The protocol type would change shape (`payload`
   becomes `paths` with em-unit path commands). The semantic and
   layout stages would not change. Migration cost: weeks, not
   months.
5. **Glyph handle eviction races.** If the engine evicts a handle
   while the renderer holds a reference, the next render fails to
   resolve. Mitigation: handles carry an `epoch`; on epoch mismatch
   the renderer skips the text and logs (this is a deterministic-
   miss case, not a render-time correctness bug).
6. **Imported KiCad text with `OUTLINE_FONT` that uses a TrueType
   not in Datum's bundled set.** Out of Phase 3 scope; Phase 1 and
   the M7-IMP-014 brief settled this as a follow-on slice. Current
   recommendation: TrueType with no font in the bundle falls back
   to the default-intent font (Inter for Annotation, Newstroke for
   Manufacturing) and the import is logged as a font substitution.

## Comparison to KiCad's Approach

KiCad is the closest peer implementation and warrants a specific
call-out.

KiCad's stack:
- **Semantic**: `EDA_TEXT` / `PCB_TEXT` / `SCH_TEXT` / `FP_TEXT`
  carry the text content, font reference, position, rotation,
  alignment, mirror, etc. Identity is preserved end-to-end.
- **Shaping**: HarfBuzz via `KIFONT::OUTLINE_FONT::Draw` (for TTF
  text) or KiCad's in-house Newstroke renderer (for stroke text).
- **Glyph contour cache**: yes, in `KIFONT::OUTLINE_FONT::m_glyphCache`.
  Keyed by glyph_id within a font instance. Contains flattened
  Bézier polylines.
- **Glyph mesh cache**: **no**. KiCad's GAL backends tessellate per
  draw call.
- **Per-text batching**: **partial**. The OpenGL GAL's
  `GAL::BeginGroup` / `EndGroup` mechanism allows grouping primitives,
  but TTF text rendering doesn't currently use it efficiently — each
  `DrawPolygon` call goes through `tesselator.cpp` afresh.
- **Atlas**: **no** for board text. Yes for the on-screen overlay
  text (status bar, etc).
- **Hit testing**: object-level via `EDA_TEXT::HitTest`, *not*
  fragment-level. KiCad gets this right.

What KiCad gets right:
- Semantic identity at every layer.
- Object-level hit testing.
- Glyph contour cache (Tier 1).

What KiCad does *not* yet do:
- Glyph mesh cache (Tier 2). Bug tracker entries `#11389` and
  `#13728` document the user-visible perf cost.
- Per-text batched draws.
- Atlas acceleration for screen-space text (PCB layer text is
  always world-space).

Where Datum can do better:
- Add Tier 2 (glyph mesh cache) — KiCad's open gap.
- Per-text batching at the protocol level (the recommended
  `BoardTextGeometry` type).
- Future Phase 3.2 atlas acceleration for zoom-out, hand-off rules
  ground-truthed against the mesh path.

Where KiCad's design is good and Datum should mirror:
- **Single shaping path** for all text: KiCad routes both stroke and
  outline text through `KIFONT`'s shaping API. Datum's
  `default_backend_for_attributes` follows the same shape and should
  continue to. (This satisfies non-negotiable constraint #5: "do not
  split imported text and native text into divergent rendering
  paths.")
- **Object-level hit testing** in the GUI layer, separate from
  geometry: Datum's `BoardTextPrimitive` (semantic) + `BoardText-
  Geometry` (render-only) split mirrors this and improves on it by
  formalizing the separation in the protocol.

## Non-Negotiable Constraints — Compliance Map

| Constraint | How the recommendation satisfies it |
|------------|-------------------------------------|
| 1. Do NOT propose falling back to stroke fonts for performance. | The recommendation keeps the dual-mode (stroke + outline) backend selection from Phase 2 unchanged. Performance comes from caching, not from fallback. |
| 2. Do NOT propose degrading visual fidelity to solve performance. | Tessellation tolerance stays at the size-relative geometric error budget already implemented in `outline.rs:outline_flatten_tolerance_nm`. Mesh quality matches contour quality. |
| 3. Do NOT propose treating every text fragment as a generic `BoardGraphicPrimitive`. | The recommendation introduces a dedicated `BoardTextGeometry` protocol type that replaces the per-fragment `BoardGraphicPrimitive` path entirely (after Phase 3.1). |
| 4. Do NOT make renderer behavior depend on KiCad `render_cache`. | The mesh path consumes Datum's own cached glyph contours from `ttf-parser` flattening. `render_cache` is not referenced anywhere in the recommended pipeline. |
| 5. Do NOT split imported text and native text into divergent rendering paths. | Both imported and native text materialize as `BoardText` semantic objects, then flow through the same Tier 1–4 caches and the same `BoardTextGeometry` protocol type. The M7-IMP-014 work (which finishes the imported normalization) feeds the same generator. |
| 6. Do NOT move the burden onto the GUI renderer to reconstruct text semantics from anonymous polygons. | The renderer never sees polygons; it sees `BoardTextGeometry` with `text_uuid`. Semantic reconstruction is structurally impossible because the polygon path is removed. |
| 7. Text object identity MUST survive through the pipeline. | `text_uuid` is carried in `BoardTextPrimitive` (semantic), `BoardTextGeometry` (render), and through the engine cache hierarchy as the Tier 3/4 key component. |
| 8. Hit testing MUST remain object-level, not fragment-level. | `board_text_hit_rect` continues to operate on `BoardTextPrimitive` (semantic). The geometry stream is render-only and never participates in hit testing. |
| 9. Datum owns the text engine. | All recommended dependencies (`ttf-parser`, `lyon`, `i_overlay`, optionally `glyphon`) are permissively licensed Rust crates. No subprocess shell-out, no GPL link, no external service dependency. The engine remains in-process, deterministic, and self-contained. |
